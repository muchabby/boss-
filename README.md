# leiting-hiring

BOSS 直聘招聘助手，基于 web-access CDP 接管浏览器，支持简历筛选、站内沟通、飞书约面三类服务。

## 前置依赖

使用前需要完成以下三项安装，每台电脑只需装一次。

### 1. Node.js 22+

下载地址：https://nodejs.org/

安装完成后验证：
```bash
node -v  # 应显示 v22.x.x 或更高
```

### 2. web-access skill

```bash
git clone https://github.com/eze-is/web-access ~/.claude/skills/web-access
```

### 3. Chrome 开启远程调试

在 Chrome 地址栏打开 `chrome://inspect/#remote-debugging`，勾选 **Allow remote debugging for this browser instance**。

> 每次重启 Chrome 后需要重新勾选。

1. 完成上方三项前置依赖
2. 打开并登录 BOSS 直聘招聘者页面（`https://www.zhipin.com/web/chat/index`）

## 核心设计原则

- **启动轻**：静默探针，环境 OK 直接进
- **防封号**：消息只走页面按钮、安全验证立即停、每日发消息上限 30 条
- **人工确认**：筛选结果给用户看，用户确认后才发消息
- **持久化**：候选人记录跨会话保留，不重复处理
- **站点经验**：BOSS DOM 路径已验证，不重复探索

## 触发词

- 「帮我筛简历」「筛一下投递」→ 进入 screening
- 「帮我联系这些候选人」「发消息」→ 进入 chat
- 「帮我约面」→ 进入 scheduling

## 更新日志

### v1.3.0（2026-05-13）

本次更新以 `leiting-hiring` 为主线，融合 `boss-hiring-assistant` 中更制度化的风控、浏览器路由和筛选覆盖规则。

**BOSS 页面统一只走 web-access**
所有 BOSS 页面读取、点击、线程切换、详情读取和发送动作统一通过 `web-access` 执行。明确不使用 `browser-use` 或其他通用浏览器 skill 抢占 BOSS 页面控制。

**截图、OCR、图像理解默认禁止**
默认坚持 DOM / CDP 路径。只有主路径失败、同一路径有限补试仍失败、已说明 token 与风控风险，并且用户明确同意后，才允许临时启用截图异常方案。

**启动前增加静默环境检查**
启动时优先静默检查 `web-access` 依赖、`/targets` 可访问性，并通过 `document.title` 做低成本登录态探针。检查通过时不打扰用户，失败时再提示具体阻塞。

**targetId 变化时自动重绑定**
当 Boss tab 刷新、重建或 `targetId` 变化时，先重新调用 `/targets` 查找 zhipin.com target，再做登录态探针，不再把 target 变化直接等同于掉登录。

**发送前加入固定 recipe 和候选人身份校验**
所有站内发送动作前都要先打开候选人线程、确认右侧面板已切到目标候选人，并在发送前和点击前做身份校验，避免把消息发给错误候选人。

**保留 leiting 的消息发送策略**
继续禁止通过 CDP/eval 向输入框写入中文；求简历、默认打招呼优先走 BOSS 页面按钮。`btn.click()` 仅作为「求简历确认弹窗」的已验证特例，不作为通用点击主路径。

**候选人来源扩展为三类**
筛选来源从主动投递 / 推荐页扩展为三类：`inbound_chat`、`recommended_feed`、`search_results`。搜索页仅在账号具备搜索权益时执行。

**推荐页和搜索页使用持久化判断本轮未处理候选人**
推荐页和搜索页不依赖页面自带“未读”标记，默认结合本地持久化记录判断哪些候选人属于本轮未处理，避免重复处理旧候选人。

**历史重扫规则补齐**
历史候选人默认不重扫。只有用户点名、新未读回复、上一轮未完成 / 待补读、候选人状态明显变化时，才重新进入处理，并记录重扫原因。

**BOSS DOM 经验补强**
明确沉淀 `[role="listitem"]`、`data-id`、`#_${dataId}`、`.conversation-box`、`.chat-top-filter` 等已验证路径，减少重复探索。

### v1.1.0（2026-04-24）

本次更新来自实际操作中发现的问题，全部针对真实踩坑场景。

**再也不会发出乱码消息**
通过 CDP 向输入框写中文会产生乱码，乱码消息会触发 BOSS 风控。现在所有消息只走页面上的按钮（求简历按钮、默认打招呼按钮），彻底杜绝乱码。

**遇到安全验证立即停止，不再继续操作**
之前安全验证弹出后操作仍在继续，存在封号风险。现在只要出现验证码或安全验证弹窗，立即停止全部操作并通知用户，等用户处理完再恢复。

**不会重复给同一个人发求简历**
发送「求简历」前会先检查聊天记录，如果已经发过、或候选人已经发来附件，直接跳过，不重复打扰。

**专升本候选人会被正确识别为大专学历**
之前只看学历标签，「专升本」字样藏在教育经历文字里会被漏掉，导致误判为本科通过筛选。现在读取完整教育经历文字，专升本正确归类为大专。

**期望岗位不再影响筛选结果**
候选人填的期望岗位与实际投递岗位不一致很常见，不应作为筛选依据。现在忽略期望岗位字段，只看实际经历和技能。

**启动时一次性授权，不再逐条弹提示**
每次操作都弹授权确认很打断节奏。现在启动时统一授权一次，本次会话内所有操作不再重复询问。

**筛选速度提升，明显不合适的快速跳过**
之前每个候选人都有固定停顿，拖慢整体速度。现在明显不符合要求的直接跳过，有潜力的看 3-5 秒，节奏更接近人工操作。

**过程中不再中途暂停等待确认**
之前每发送 3 人就暂停等用户确认，打断操作流。现在全程连续执行，只在筛选汇报后才等用户决策。


### v1.2.0（2026-04-28）

**筛选前先读JD出画像，用户确认后再开始筛人**
之前直接开始筛选，没有给用户确认筛选标准的机会。现在每次筛选前会先读取所有在招岗位的JD，自动生成简洁画像（硬门槛+直接淘汰条件），一次性展示给用户确认或修改，确认后才开始筛人。避免用过期标准筛人，也让用户每次都能校准。

**全局授权一次记住，不再重复询问**
用户一次性授权后，本次会话内所有浏览器操作直接执行，不再弹出任何授权确认。之前授权状态没有被正确记住，导致每次操作都打断用户。

**边看边发，不再二次筛选**
之前的流程是先看一遍汇报结果，再回头找到候选人发求简历，效率低下。现在看到合适的候选人立即发求简历，最后再统一汇报结果，减少重复操作。

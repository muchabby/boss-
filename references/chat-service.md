# 站内沟通服务

## 目标

向通过筛选的候选人发送 BOSS 站内消息。

只负责：索要附件简历、确认意向、收集可约时间。
不负责：筛选简历、飞书约面。

---

## 进入条件

必须满足以下两个条件才能进入：

1. 用户明确指定了本轮要沟通的候选人（来自筛选汇报后的确认）
2. 当前操作在时间窗口内

不要自行决定哪些候选人进入沟通，必须等用户确认。

---

## 消息策略

- 默认使用 BOSS 系统自带话术（「你好啊，可以聊一聊~」等）
- 不自行生成消息内容
- 用户有定制化需求时，由用户提供消息文本，skill 负责发送

---

## 发送流程

对每个待沟通候选人，按以下顺序执行：

### 1. 打开候选人线程

```js
// 定位候选人卡片
// 使用 data-id 精确定位，不用位置索引
clickAt('#_{candidate_data_id}')
```

等待右侧面板切换到目标候选人，确认姓名一致后继续。
线程未成功打开时，最多重试 2 次，仍失败则跳过并记录。

### 2. 候选人身份三次校验

- 切换线程后校验一次
- 写入消息前校验一次
- 点击发送前校验一次

任意一次不一致，立即停止，进入 `paused_for_wrong_candidate`。

### 3. 写入并发送消息

输入框 selector：`.chat-container-private [contenteditable]`
发送按钮 selector：`.chat-container-private .submit`

点击发送优先级：
1. `clickAt` — 模拟真实鼠标事件
2. CDP `Input.dispatchMouseEvent`
3. DOM `element.click()` 仅作一次性兜底

### 4. 发送成功确认

- 输入框内容清空 → 发送成功
- 最后一条消息出现「送达」标记 → 二次确认

### 5. 发送间隔

每条消息发送后随机停顿 3s-8s，不允许连续发送。
每发送 3 人暂停，等待用户确认继续。

---

## 附件简历闭环

发送「求简历」消息后，必须跟进到闭环：

1. 记录状态 `attachment_requested`
2. 后续扫描时优先检查该候选人是否已回复附件
3. 候选人发回附件后，检查是否出现「同意/接收」按钮
4. 点击接收，更新状态为 `attachment_received`
5. 接收完成后，读取附件内容，更新筛选判断

---

## 每日发送上限

每日主动发消息不超过 30 条（跨会话累计）。
达到上限时停止发送，通知用户。
上限计数写入持久化记录，次日自动重置。

---

## 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| 线程打开失败（3次） | 跳过该候选人，记录 `thread_open_failed` |
| 候选人身份不一致 | 立即停止，进入 `paused_for_wrong_candidate` |
| 发送失败（主路径+备用路径均失败） | 跳过，记录 `send_failed`，通知用户 |
| 出现验证码/异常弹窗 | 立即停止所有发送，通知用户 |
| 每日上限达到 | 停止发送，通知用户 |
